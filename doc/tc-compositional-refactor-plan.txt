GOAL
----
Move tc from imperative-IO style toward a more compositional Lean 4 style
by applying 9 Haskell-inspired patterns. Each task is scoped to run as its
own `/feature` workflow (worktree → plan → approve → implement → PR).

Tasks that touch the same files are grouped so parallel worktrees don't
collide at merge time.

--------------------------------------------------------------------------------
 KNOWN RISKS / CONFLICTS
--------------------------------------------------------------------------------

  * B vs C both rewrite appMain / App/Common.lean.
    → Whichever merges second must rebase. Suggest merging C first (smaller
      surface) then B.

  * A may touch App/Common.lean line 94 (nested update).
    → If A lands first, B and C rebase trivially. If B or C lands first,
      A's lens edit at that line needs to re-target the new code shape.

  * D vs E both touch Session.lean area.
    → Keep D out of Session.lean entirely. D only edits Runner.lean and
      creates Tc/Pipe.lean. E has Session.lean to itself.

  * F (Filter.lean) may clash with A (which also touches Filter.lean:19
    for the nested-update fix).
    → A's edit is one line; do A first, then F rebases trivially.

  * Each refactor requires `lake build` to verify. Build time may exceed
    single-command timeouts for heavy groups (B, C, H). Agents should run
    build in background or split into multiple bash calls.
